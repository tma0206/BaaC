# IT トライブ趣意書（草案）：植物をプログラムとして見る

## なぜ IT トライブ向けなのか
ソフトウェアと盆栽は、どちらも「時間をかけて状態を良くする」活動です。バグは突然表面化し、原因は環境・履歴・小さな積み重ねにあります。IT 系の学習者には、**仮説・検証・ロールバック** の思考が既にあることが多く、盆栽の「季節・樹種・個体差」という制約の中での最適化と相性が良い、という立場を取ります。

## 植物をプログラムにたとえるときのルール
- **入力**：光、水、温度、風通し、肥料、剪定という「パラメータ」
- **出力**：萌芽、伸長、色、根の健全性、花芽形成などの「挙動」
- **ログ**：去年の剪定、植え替え時期、鉢サイズ、置き場所の変更（履歴が推論の質を上げる）
- **レガシー**：長年の育成史は「既存コード」。大きな書き換えにはコストとリスクがある

## このプロダクトが目指すこと
- 迷いを減らす：**今日の一手**と**確認事項**を返す
- 断定の義務を限定：樹種・環境が不明なときは質問を増やす
- 改善ループ：`issues` とテスターからのフィードバックで Oracle のナレッジとプロンプトを更新する

## 非目標
- 植物の確定診断や治療保証
- 投資・転売を煽ること
